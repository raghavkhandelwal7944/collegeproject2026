FIREWALL LLM CODEBASE EXPLANATION
=================================

Purpose of this file
--------------------
This document explains the main logic of the entire codebase in plain English.
It focuses on:
1. What each major backend and frontend file does
2. The main Python, TypeScript, and React functions/components used
3. The full request workflow from login to chat response
4. How each security policy works
5. How chat history works across MongoDB and the React UI
6. How MySQL, MongoDB, Redis, FastAPI, Next.js, and React work together


============================================================
1. HIGH-LEVEL IDEA OF THE PROJECT
============================================================

This project is a secure AI chat application.

The main idea is:
- A user logs into a web dashboard
- The user sends prompts through a chat interface
- Before the prompt reaches the main language model, it passes through a firewall pipeline
- That pipeline can:
  - detect prompt injection or jailbreak attempts
  - detect and redact PII
  - store token mappings in Redis
  - check semantic cache in Redis
  - block dangerous code-style prompts
  - send only safe prompts to the main model
  - restore redacted data in the final answer when allowed
- The app also stores:
  - audit logs in MySQL
  - conversation history in MongoDB
  - temporary token vault and semantic cache in Redis

So this is not just a chatbot.
It is a chatbot + AI firewall + audit layer + policy engine.

2. MongoDB

5. Gemini

============================================================
3. TOP-LEVEL PROJECT FLOW
============================================================

End-to-end user flow:
1. User signs up or logs in from the Next.js frontend
2. Next.js auth route talks to FastAPI backend
3. Backend returns JWT token
4. Next.js stores JWT inside an httpOnly cookie named token
5. Protected frontend pages are allowed only if token cookie exists
6. User opens Playground page and sends a prompt
7. Frontend sends prompt to /api/proxy/api/v1/chat
8. Next.js proxy route forwards request to FastAPI backend with Authorization: Bearer <token>
9. FastAPI chat router runs firewall pipeline
10. If safe, main LLM responds
11. Backend stores conversation and logs
12. Frontend displays final response

4.2 backend/database.py
------------------------------------------------------------

This is the persistence layer.
It handles MySQL and MongoDB operations.

Data split by storage type
--------------------------
MySQL stores:
- users
- logs
- user_policies

MongoDB stores:
- conversations
- session-based message turns

Main functions
--------------
1. get_mongo_connection()
   Creates MongoDB connection and returns the DB object.

2. generate_session_title(user_message)
   Uses Gemini to produce a short title for a new chat session.
   If Gemini fails, it falls back to the first 40 chars of the first message.

3. save_conversation(username, user_message, bot_response, session_id, session_title)
   Saves one full conversation turn to MongoDB.
   Stored fields:
   - username
   - user_message
   - bot_response
   - session_id
   - session_title
   - timestamp

4. get_user_conversations(username, limit=50)
   Returns recent conversations for a user.

5. list_user_sessions(username)
   Important for chat history.
   Uses Mongo aggregation to group messages by session_id.
   Returns:
   - session_id
   - title
   - last_message timestamp
   - message_count

6. get_session_messages(username, session_id)
   Important for restoring old chats.
   Reads one session from MongoDB and converts stored turns into:
   - {role: 'user', content: ...}
   - {role: 'assistant', content: ...}
   sorted oldest-first

7. get_db_connection()
   Returns MySQL connection.
   If DB does not exist, it tries to create it.

8. create_database()
   Creates the MySQL database if missing.

9. init_db()
   Creates the MySQL tables if missing:
   - logs
   - users
   - user_policies
   Also migrates missing user columns.

10. get_user_policies(username)
    Reads user policy flags from MySQL.
    If no row exists, creates default row:
    - aggressive_pii = False
    - semantic_cache = True
    - code_block = False

11. set_user_policies(username, aggressive_pii, semantic_cache, code_block)
    Upserts policy settings for a user.

12. get_conversation_history(username, limit=20, session_id=None)
    Very important for chat memory.
    Reads MongoDB conversations and returns model-ready context.
    If session_id is provided, history is scoped only to that session.
    This is what fixed context bleed between old and new chats.

13. create_user(username, hashed_password, ...)
    Inserts user into MySQL.

14. get_user(username)
    Reads user row for authentication.

15. log_request(user_input, blocked, violation_type)
    Stores audit trail in MySQL logs table.

16. get_recent_logs(limit=20)
    Returns latest audit logs.

17. get_stats()
    Computes:
    - total requests
    - total blocked
    - percentage blocked

------------------------------------------------------------

This file is the Layer 0 firewall logic.
It handles prompt injection detection with Gemini.

Main functions
--------------
1. detect_injection(prompt)
   Sends the prompt to Gemini with a system instruction that says:

------------------------------------------------------------
4.6 backend/services/presidio_service.py
------------------------------------------------------------

This is Layer 1: PII detection and anonymization.

Main purpose
------------
It scans a prompt for sensitive entities and replaces them with deterministic tokens.
Example:
- "John Smith" -> <PERSON_a3f2c1>

Main constants/helpers
----------------------
1. _TARGET_ENTITIES
   The entity types currently scanned:
   - PERSON
   - EMAIL_ADDRESS
   - PHONE_NUMBER
   - CREDIT_CARD

2. _SELF_NAME_RE
   Regex fallback for lower-case self-introduction phrases like:
   - "my name is raghav"

3. _CARD_RE
   Regex fallback for potential payment card strings.

4. _make_token(entity_type, original_text)
   Creates deterministic token using SHA-256 hash prefix.
   Same input -> same token.

5. _looks_like_payment_card(text)
   Uses Luhn algorithm to validate card-like strings.

6. _build_fallback_results(prompt, existing)
   Adds extra detections only in aggressive mode.
   This was added so lower-case names and some card numbers are still caught.

Main class
----------
PresidioService

Important methods
-----------------
1. __init__()
   Registers lazy service; does not fully load heavy NLP immediately.

2. _ensure_loaded()
   Loads AnalyzerEngine and AnonymizerEngine on first real use.

3. scan(prompt, aggressive=False)
   This is the main PII scan function.

   Workflow inside scan():
   a. Presidio Analyzer finds entities
   b. If aggressive=True, regex fallback results are added
   c. Results are sorted
   d. If nothing found, return original prompt
   e. Build token map for each span
   f. Configure anonymizer operators
   g. Replace spans with tokens
   h. Return ScanResult with anonymized text + entity list + duration

Main internal data type
-----------------------
ScanResult
Fields:
- anonymized_text
- entities
- duration_ms

Why this file matters
---------------------
This is what makes corporate-safe prompting possible, because the model can receive a redacted prompt instead of raw confidential data.

Main class

Main methods

4.8 backend/services/redis_service.py
------------------------------------------------------------

This is Phase 3 storage logic.
It has two big jobs:
1. Token Vault
2. Semantic Cache

Token Vault concept
-------------------
When PII is redacted, the backend keeps a temporary map like:
- <PERSON_a3f2c1> -> John Smith
- <EMAIL_123abc> -> john@example.com

This is stored in Redis so the final answer can be restored later.

Semantic Cache concept
----------------------
For similar prompts, avoid calling the main model again.
Store:
- embedding vector
- anonymized LLM response

Main class
----------
RedisService

Main methods
------------
1. store_vault(session_id, token_map)
   Stores token->original text mapping in Redis hash with TTL.

2. restore_tokens(session_id, text)
   Replaces anonymized tokens in model response with original text.
   Used right before sending response to the user.

3. get_cached_response(embedding, threshold=None)
   Looks through cached Redis entries.
   Computes cosine similarity.
   Returns cached response if best score >= threshold.

4. store_cache_entry(embedding, response)
   Stores a new semantic cache entry.

Design note
-----------
All public methods fail safely.
If Redis is down:
- vault store becomes no-op
- restore returns original text unchanged
- cache lookup becomes miss
- cache store becomes no-op

This keeps the app usable even without Redis.

4.11 backend/routers/chat.py
------------------------------------------------------------

This is the most important backend file.
It implements the full firewall pipeline.

Main helper
-----------
1. _storage_safe_prompt(raw_prompt, anonymized_prompt, aggressive_pii)
   Decides what form of the prompt is safe to store.
   If aggressive_pii is on, store anonymized prompt.
   If off, store raw prompt.

Main endpoint
-------------
2. chat(request, current_user)
   POST /api/v1/chat

This function contains the full workflow.

Detailed request workflow inside chat()
---------------------------------------

Step 1: Read user and raw prompt
- username comes from JWT dependency
- raw_prompt comes from request.prompt

Step 2: Load user policy flags from MySQL
- aggressive_pii
- semantic_cache
- code_block

Step 3: Prepare default scan result
If PII policy is off, result defaults to:
- anonymized_text = raw prompt
- entities = []
- duration = 0

Step 4: Aggressive PII policy pre-scan
If aggressive_pii is enabled:
- Presidio scan runs immediately with aggressive=True
- storage_prompt becomes anonymized text
- model_prompt becomes anonymized text

This ensures storage and model both get safe text.

Step 5: Code execution block policy
If code_block is enabled:
- regex checks raw prompt for patterns like:
  exec, eval, os.system, subprocess, shell, powershell, bash, DROP TABLE, DELETE FROM, etc.
- if matched:
  - log_request(... blocked=True ...)
  - return HTTP 403

Step 6: Injection detection (Layer 0)
Runs detect_injection(raw_prompt)
- if Gemini says malicious:
  - write audit log
  - return 403

Step 7: PII gate logic
If aggressive_pii is on:
- use anonymized prompt for storage and model
If off:
- skip redaction entirely

This is important:
PII policy is a true on/off switch.
When off, no redaction is applied.

Step 8: Determine violation_type
- "PII" if entities found
- "None" otherwise
This gets stored in audit logs.

Step 9: Create chat_session_id and vault_id
- chat_session_id groups a full conversation
- vault_id groups one request's token vault entry

Step 10: Token Vault write
If Redis exists, aggressive_pii is on, and entities exist:
- store token map in Redis

Step 11: Semantic cache lookup
If Redis exists and semantic_cache is enabled:
- embed model_prompt
- look for cached response

If cache hit:
- optionally restore tokens using vault
- log request as allowed
- save conversation to MongoDB
- return ChatResponse immediately
- skip gatekeeper and main model

Step 12: Gatekeeper model (Layer 2)
- call check_with_gatekeeper(model_prompt)
- if unsafe:
  - log blocked request with category
  - return 400

Step 13: Load conversation history for current session
- get_conversation_history(username, session_id=chat_session_id)
- this returns only same-session messages

Step 14: Merge frontend temporary turns
The frontend can send messages in request.messages.
These are client-side in-memory turns not yet persisted.

combined_history = client_history if client_history else history

Step 15: Main model call (Layer 3)
- call_main_llm(model_prompt, history=combined_history)

Step 16: Cache store
If semantic cache enabled:
- store new embedding + raw LLM response in Redis

Step 17: De-anonymize final response
If aggressive_pii and Redis vault exist:
- restore original text in response
Else:
- final_response = llm_response

Step 18: Save conversation to MongoDB
Saves:
- storage_prompt
- llm_response
- session_id
- session title if first message in session

Note: Mongo stores llm_response, not final_response.
That means storage remains safer than user-facing view.

Step 19: Save audit log to MySQL
- blocked=False
- violation_type based on PII detection

Step 20: Return ChatResponse
Contains both:
- llm_response (raw model output, possibly still tokenized)
- final_response (restored text for user)


============================================================
5. POLICY SYSTEM EXPLANATION
============================================================

There are 3 main policies.

------------------------------------------------------------
5.1 Aggressive PII Redaction
------------------------------------------------------------

Stored in MySQL as aggressive_pii.
Frontend name: aggressivePii.

When ON:
- backend scans prompt with Presidio + fallback regexes
- prompt sent to model is anonymized
- prompt stored in logs/history is anonymized
- token map is stored in Redis
- final response can be de-anonymized for user display

When OFF:
- no PII scan is applied
- raw prompt is sent to model
- raw prompt is stored
- no token vault flow is used

Important detail:
This was intentionally designed as a strict true on/off gate.


------------------------------------------------------------
5.2 Semantic Cache
------------------------------------------------------------

Stored in MySQL as semantic_cache.
Frontend name: semanticCache.

When ON:
- backend creates embedding from prompt
- checks Redis cache for similar prompt
- if hit, returns cached response and skips LLM
- after new LLM response, stores embedding+response in cache

When OFF:
- no embedding generation
- no Redis cache lookup
- no cache storage
- every request reaches the model pipeline


------------------------------------------------------------
5.3 Code Execution Block
------------------------------------------------------------

Stored in MySQL as code_block.
Frontend name: codeBlock.

When ON:
- backend regex scans raw prompt for dangerous code or destructive commands
- if matched, request is blocked before expensive model work

When OFF:
- this extra regex block does nothing

Examples of blocked patterns:
- exec
- eval
- os.system
- subprocess
- shell=
- powershell
- bash
- DROP TABLE
- DELETE FROM
- TRUNCATE

6. CHAT HISTORY IMPLEMENTATION (VERY IMPORTANT)
============================================================

This project has session-based chat history.

Goal
----
- New chat should start empty
- Old chat should preserve only its own history
- Old chats should not contaminate new ones

How it is implemented
---------------------

Backend side
------------
1. Every chat request has chat_session_id
2. If frontend sends no session ID, backend creates new UUID
3. save_conversation() stores each turn in MongoDB with session_id
4. list_user_sessions() groups history by session_id
5. get_session_messages(username, session_id) rebuilds one old session for the UI
6. get_conversation_history(username, session_id=...) returns only that session for model context

This is what prevents context bleed.

Frontend side
-------------
The main logic is in frontend-react/lib/chatStore.ts and PlaygroundChat.tsx.

chatStore.ts
------------
This is a module-level store, not React state inside one component.
That means it survives page navigation.

Main chatStore functions
------------------------
1. getMessages()
   Returns current message array.

2. isSeeded()
   Whether history has already been loaded.

3. markSeeded()
   Marks that initial session state is set.

4. getSessionId()
   Returns current active session ID.

5. getLoading() / setLoading(v)
   Keeps loading state even if page unmounts.

6. setMessages(msgs)
   Replaces current chat with loaded session messages.

7. push(...msgs)
   Appends new chat messages.

8. newSession()
   Starts new chat.
   It:
   - clears messages
   - sets seeded=true
   - resets loading=false
   - generates fresh UUID session ID

9. clearChat()
   Used on logout.
   Clears everything and resets state.

PlaygroundChat.tsx history logic
--------------------------------
1. On initial mount:
   - it does NOT auto-load all old messages anymore
   - it only marks the store as seeded
   This keeps a new chat empty

2. refreshSessions()
   Calls backend getChatSessions() to populate history dropdown

3. handleNewChat()
   Calls chatStore.newSession()
   This creates a completely fresh conversation

4. handleLoadSession(session)
   When user clicks an old session:
   - fetches getSessionMessages(session.session_id)
   - converts them into ChatMessage[]
   - loads them into chatStore.setMessages()
   - sets currentSessionId
   - sets activeSessionTitle

5. handleSubmit(e)
   Before sending message:
   - gets current session ID from loaded session or current store
   - builds priorTurns from existing in-memory messages
   - sends prompt + priorTurns + sessionId to backend

6. Backend then uses that same session ID to scope Mongo history retrieval

Result
------
- New chat = no old history
- Old chat = only its own history
- Model context is isolated per session

============================================================
8. HOW AUTHENTICATION WORKS
============================================================

1. User submits login form in LoginForm.handleSubmit()
2. frontend lib/api.ts login() calls /api/auth/login
3. Next login route forwards credentials to FastAPI /token
4. FastAPI verifies password and returns JWT
5. Next route stores token in httpOnly cookie
6. proxy.ts checks cookie for protected pages
7. app/api/proxy/[...path]/route.ts attaches token as Bearer auth to backend calls

Why this design is good
-----------------------
- JWT is not exposed directly to client-side JS
- browser keeps auth in cookie
- Next.js server routes handle backend communication safely


============================================================
9. HOW DATA MOVES THROUGH THE SYSTEM
============================================================

Login data
----------
Browser -> Next auth route -> FastAPI /token -> JWT -> httpOnly cookie

Chat data
---------
Browser -> Next proxy route -> FastAPI /api/v1/chat
-> injection detection
-> optional PII redaction
-> optional Redis vault/cache
-> gatekeeper model
-> main model
-> MongoDB save
-> MySQL audit log
-> response back to browser

Policy data
-----------
Browser -> Next proxy route -> FastAPI /api/v1/policies -> MySQL user_policies

Traffic page data
-----------------
Browser -> Next proxy route -> FastAPI /activity_logs -> MySQL logs

Dashboard stats data
--------------------
Browser -> Next proxy route -> FastAPI /stats -> MySQL logs aggregation

Old session load
----------------
Browser -> Next proxy route -> FastAPI /history/{session_id} -> MongoDB conversations


============================================================
10. MOST IMPORTANT DESIGN DECISIONS
============================================================

1. Shared services are initialized once in FastAPI lifespan
   Good for performance and memory reuse.

2. Policy system is per-user and stored in MySQL
   This makes rules personalized and persistent.

3. Chat history is scoped by session_id
   This prevents old chats from polluting new chats.

4. Frontend chat state is module-level, not page-local only
   This preserves user experience during navigation.

5. Redis features degrade gracefully
   The app still works if Redis is unavailable.

6. Prompt storage uses anonymized prompt when PII policy is enabled
   This keeps storage safer than raw prompts.

7. UI risk scoring is separate from backend enforcement
   The frontend score is just a visual hint.
   Real security decisions happen in backend/chat.py.

11. IF YOU WANT TO EXPLAIN THIS PROJECT TO SOMEONE QUICKLY
============================================================

Short explanation
-----------------
This is a secure AI chat platform where all prompts pass through a firewall before reaching the LLM. The firewall can detect prompt injection, redact PII, block unsafe code-related prompts, reuse cached answers for similar prompts, store audit logs, and keep chat history separated by session.

One-line academic summary
-------------------------
This codebase implements a policy-driven LLM firewall using FastAPI, Next.js, MySQL, MongoDB, Redis, Presidio, and an OpenAI-compatible inference server to provide secure, auditable, session-aware enterprise AI interactions.
